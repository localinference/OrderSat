# TypeScript Package Walkthrough

This module turns the exported runtime artifacts from `06`, `07`, and `08`
into a self-contained TypeScript source package under
`src/09_autogenerated_typescript_project`.

For each language that has all required runtime bundles, it emits:

- a tokenizer loader
- a WebGPU model loader
- a WASM model loader
- model metadata for inference

## Entrypoint

Run:

```powershell
node cli/09_generate_typescript_project/index.js
```

The entrypoint is [index.js](C:/Users/jorts/OrderSaT/cli/09_generate_typescript_project/index.js).

## High-Level Flow

The generator does this, in this order:

1. Read available language directories under:
   - `src/06_fp32_export_onnx_models`
   - `src/07_mixed-fp16_gpu_onnx_models`
   - `src/08_uint8_cpu_onnx_models`
2. Keep only languages that exist in all three roots.
3. For each shared language:
   - read `tokenizer.model` from `06`
   - read `model.mixed-fp16.onnx` from `07`
   - read `model.uint8.onnx` from `08`
   - read `config.json` from `06`
4. Compress each binary artifact with `@sovereignbase/bytecodec`.
5. Encode the compressed bytes as base64 string literals.
6. Generate `src/09_autogenerated_typescript_project/models/{language}/index.ts`.
7. Generate the root re-export file at
   `src/09_autogenerated_typescript_project/index.ts`.

## What It Reads

For `eng`, this module reads:

- `src/06_fp32_export_onnx_models/eng/tokenizer.model`
- `src/06_fp32_export_onnx_models/eng/config.json`
- `src/07_mixed-fp16_gpu_onnx_models/eng/model.mixed-fp16.onnx`
- `src/08_uint8_cpu_onnx_models/eng/model.uint8.onnx`

## What It Writes

For `eng`, this module writes:

- `src/09_autogenerated_typescript_project/models/eng/index.ts`
- `src/09_autogenerated_typescript_project/index.ts`

The language module exports:

- `tokenizerENG(): Promise<Uint8Array>`
- `gpuModelENG(): Promise<Uint8Array>`
- `cpuModelENG(): Promise<Uint8Array>`
- `modelInfoENG`

`modelInfoENG` currently exposes:

- `bosTokenId`
- `eosTokenId`
- `padTokenId`
- `maxSourcePositions`
- `maxTargetPositions`

## Why It Reads Metadata From `06`

`06` is the canonical selected export bundle for a language.

That means:

- `06` already chose the better trained format
- `06/config.json` holds the authoritative model contract
- `07` and `08` are runtime-specific optimizations of that same selected model

So `09` takes:

- tokenizer + metadata from `06`
- GPU bytes from `07`
- CPU bytes from `08`

This keeps the package aligned with the single selected model, instead of making
the consumer guess which tokenizer or limits belong to which runtime artifact.

## Why The Artifacts Are Embedded

The generated TypeScript package is meant to be portable and easy to consume
locally.

So `09` embeds the binaries directly into TypeScript source as compressed
base64, then exposes async loader functions that decompress them on demand.

That avoids runtime file-path coordination in the consumer package while still
keeping the stored source smaller than raw uncompressed byte literals.

## Design Intent

This module is built around these rules:

1. Only package languages that have a complete runtime set.
2. Treat `06` as the canonical source of model identity and metadata.
3. Expose one simple loader surface per language.
4. Make the consumer import the real runtime contract instead of guessing token
   ids or position limits.
